# Emoji → Iconoir sweep — Retrospective

_Closed: 2026-07-16_

## What shipped

- **Sprint 1.** ✅ MERGED 2026-07-12, frontend [#235](https://github.com/danybgoode/miyagisanchezcommerce/pull/235) → squash `064f036`. Mechanical sweep of 86 files (the 2026-07-05 scope doc's audit had gone stale — regenerated fresh, up from 79) replacing bare emoji rendered as UI chrome with Iconoir `<i>` elements across the storefront, seller shell, and buyer flows. Plus the CI guard (`lib/emoji-guard.ts` + `e2e/emoji-guard.spec.ts`), modeled directly on the existing raw-color guard, hard-gating 69 fully-clean files and documenting the rest (5 voice `🎉` exceptions, 10 plain-string-data deferrals) with a reason.
- **Three fast-follow PRs**, all found by Daniel's post-merge visual pass, none caused by this sweep — a pre-existing, unrelated class of bug the sweep's own verification method happened to be perfectly positioned to catch:
  - [#239](https://github.com/danybgoode/miyagisanchezcommerce/pull/239) — the 2 broken classes flagged in #235's own PR description (`iconoir-newspaper`, `iconoir-cancel`), plus 2 more found doing the same diff (`iconoir-bag-plus`, `iconoir-lightning-bolt`).
  - [#240](https://github.com/danybgoode/miyagisanchezcommerce/pull/240) — Daniel spotted broken icons in the seller-shell and admin left-nav rails; found 3 more (`iconoir-pricetags`, `iconoir-text-box-edit`, `iconoir-toggle-on`) plus a bonus British-spelling typo (`iconoir-colour-filter`) once the check was broadened from `.tsx`-only to `.ts`+`.tsx`.
  - [#260](https://github.com/danybgoode/miyagisanchezcommerce/pull/260) — Daniel spotted broken icons on the `/vende` and `/vende/creadores` marketing pages; found 5 broken classes in `locales/{es,en}.json`, the bilingual copy dictionary — outside `app/`+`lib/`+`components/` entirely, missed until the check was broadened again to include `.json`.

**11 broken Iconoir classes found and fixed total** across the four PRs. Every one predates this epic — `app/layout.tsx` loads Iconoir unpinned from `iconoir-icons/iconoir@main` on jsdelivr, and an unknown class silently resolves its CSS `mask-image` to nothing rather than erroring, so nothing short of a class-by-class diff against the real bundle would ever surface them. Codex cross-review caught one real (cosmetic) formatting bug pre-merge on #235; every other cross-review pass came back clean.

## What went well

- **Verifying every icon class against the actually-loaded bundle, not memory or convention, from the very first commit.** The plan explicitly called this out before any code was written (`app/layout.tsx` loads an unpinned `@main` CDN reference — a real drift vector) and it's what let the fast-follow work start immediately from a working method instead of guessing.
- **A staged, honest hard gate instead of an all-or-nothing one.** `enforcedSweptPaths` only asserts zero-tolerance on the files verified fully clean; the 10 files with a genuinely deferred plain-string emoji are visible-but-advisory, matching the exact incremental-adoption shape the raw-color guard already established. The guard shipped honest about its own limits rather than either over-claiming coverage or blocking merge on unrelated pre-existing debt.
- **Treating a user's small correction as a signal to re-run the whole verification, not just fix the one instance reported.** Each of the three fast-follow rounds started from "here's one broken icon" and ended with 2-5x more found by re-running the *same* diff with *wider scope* — the fix was never just the reported instance.

## What we learned

- **A "verify against the real source" check is only as complete as its own file-type scope, and that scope needs to be argued for, not assumed.** The first pass scanned `.tsx` only (matching the sweep's own scope). The second pass had to broaden to `.ts` before it caught the seller-nav/admin-nav bugs, because both live in a plain `lib/*.ts` registry, not a component. The third pass had to broaden again to `.json` before it caught the marketing-page bugs, because that copy lives in `locales/*.json`, entirely outside `app/`+`lib/`+`components/`. Three rounds, three scope gaps, each only found because a human actually looked at the rendered page — the automated check never would have found the `.json` instances on its own until told to look there. **Generalizable: when building a "does X reference a real Y" verification, explicitly enumerate every file type/location X can legally appear in before declaring the check complete — don't infer completeness from "I checked the file types the current task touched."**
- **An unknown CSS icon class is a silent failure mode, not a loud one — which means the blast radius of never checking is invisible until someone looks at the rendered page.** Nothing in `tsc`, `next build`, or the Playwright suite catches a typo'd Iconoir class, because `<i className="iconoir-typo" />` is valid JSX referencing a valid (if pointless) CSS selector that simply matches nothing. This is a good candidate for a permanent guard (a static list-of-valid-classes check, mirroring `emoji-guard.ts`'s own shape) rather than relying on someone noticing a blank space in the UI — noted as a gap below, not built this epic (out of the sweep's own scope, and the four fast-follow PRs already fixed every instance found by hand).
- **Reporting a finding as "a heads-up, not a fix in this PR" in a PR description works as a paper trail, but doesn't substitute for someone actually acting on it.** #235's own description named `iconoir-newspaper`/`iconoir-cancel` as a known pre-existing gap 4 days before #239 fixed them — the note did its job (nothing was lost, the fast-follow found it immediately via `git log`/PR search), but it still took an explicit "please fix as fast follow" ask, not an automatic trigger, to close it. A tracked follow-up item (seed, task, or even a one-line TODO with an owner) travels better than a PR-description mention alone when the fix isn't happening in the same session.

## Gaps / follow-ups

- **No permanent guard against a broken Iconoir class exists yet.** All 11 known instances are fixed, but nothing stops a 12th from being introduced (a typo, or the CDN bundle dropping a class on a future `@main` update — it's unpinned). A `lib/iconoir-class-audit.ts` in the same shape as `emoji-guard.ts` (fetch/cache the real bundle's class list once, assert every static `iconoir-[a-z0-9-]+` reference in `app/`+`lib/`+`components/`+`locales/` resolves to a real class) would close this permanently. Not built this epic — flag as a seed for `00-ideas/` if worth prioritizing; the CDN being unpinned (`@main` instead of a pinned tag/SHA) is the more fundamental fix and probably belongs in the same follow-up.
- **10 of the 79 originally-swept files still carry emoji the mechanical pass couldn't convert** (a plain TS string with no separate icon slot at the render site — `STATUS_META.message` tables, toast strings, button-label config objects). Converting these needs a data-model change (split into `{ icon, label }` + update the render call site), not a mechanical swap. Documented per-file in `lib/emoji-guard.ts`'s comments; not scheduled.
- Unicode arrow glyphs (`← → ↑ ↓ ↗`) used as navigational wayfinding were deliberately scoped out of the whole epic (a different, more pervasive convention than colorful emoji) — still true everywhere in the app, not a gap, a boundary.
