## The reframe that unifies four of the five (read this first)
The blog's core lesson is a **separation of concerns we can adopt wholesale**:

- **Skills = the reusable procedure** (a folder: prompt + scripts + gotchas + templates). They do the work.
- **Routines = a trigger + a cloud session** (schedule / GitHub event / API). They *invoke* skills; they are
  not where the logic lives.

So your "routines" bullets and your "specialized skills" bullet are **two views of one system**. The blog's
**9 skill categories map almost 1:1 onto your asks**, which is the backbone of the whole portfolio:

| Blog category | Your ask | Build as |
|---|---|---|
| 4 · Business process / team automation | daily standup, weekly exec/retro | skills `standup-post`, `weekly-recap` (delta-only, keep a log to diff) |
| 7 · CI/CD & deployment | PR babysitting, build-order fix | skills `babysit-pr`, `build-order-sync` |
| 9 · Infrastructure operations | Vercel branch/preview pruning | skill `vercel-prune` (wraps the existing script) |
| 5 · Code scaffolding & templates | "we do many already — audit for gaps" | audit + extract the groom scaffolder + any missing scaffolds |
| 8 · Runbooks | incident runbooks | skills `<service>-debugging`, `oncall-runner` |
| 6 · Code quality & review | already have it | Routine A + `cross-review.mjs` (no new work) |
| 1/2/3 · Library-ref / verification / data | future gaps | note for later; not in this brain-dump |

**Design rule going forward:** put the logic in a **committed, reviewable skill**; let a **routine** (or a
hook, or a human `/verb`) trigger it. This keeps everything version-controlled, testable, and portable —
and avoids two systems doing the same job (the mistake the routines spike explicitly guarded against).

## The five initiatives — classification, buckets & shape

### Initiative A — Doc & memory noise sweep + rolling maintenance
- **Class:** Chore (small epic — one-time sweep + a standing maintenance mechanism). **Stage-2.5:**
  *light-enhancement* (the rolling half already exists as Routine C + the `consolidate-memory` skill; the
  one-time sweep is genuinely new work but bounded).
- **Real target (corrected):** the *repo* team-memory is tiny (`apps/miyagisanchez/memory/MEMORY.md` = 3
  lines). The over-budget docs are **`LEARNINGS.md` (1,125 lines / 114 KB)** and the **`README.md` poster
  (114 KB)** — both loaded at session start — plus the seven `.claude/context/*.md` (~1,260 lines). That's
  the sweep surface. (Confirm which "team memory" agents flag as over-budget — repo memory vs the
  `~/.claude/.../memory/` store — before scoping; both are cheap to check.)
- **In v1:** a de-noise pass on `LEARNINGS.md` + `README.md` (dedupe, prune stale, tighten) with a diff for
  your review; a **rolling maintenance skill** (`doc-hygiene`) the existing weekly Routine C can call so the
  files never re-bloat; keep the `consolidate-memory` skill in the loop for the memory store.
- **Out of v1:** restructuring the macro-section epic docs; any code/`tasks/` edits; auto-applying prunes
  (proposals + diff only — status/content changes stay a human merge, matching the HYGIENE-REPORT rule).
- **Risk:** low (docs only). **Acceptance:** session-start token cost of the always-read set drops
  measurably; no durable learning lost (diff reviewed); a repeatable skill exists so drift doesn't return.

### Initiative B — Ops reporting & nightly-fix routines
- **Class:** Chore (epic). **Stage-2.5:** *light-enhancement* — the routine rail, Telegram, prune + build-
  order scripts all exist; the **skills** that produce the standup/recap and wrap prune/build-order/PR-babysit
  are new but thin.
- **In v1 (build as skills, trigger via routine/schedule):** `standup-post` (daily → Telegram: overnight PR
  activity, CI/smoke status, build-order drift, open-PR state, stale Vercel previews — delta only, keeps a
  log); `weekly-recap` (weekly → Telegram: merged PRs, closed epics, deploys, a retro digest);
  `build-order-sync` (nightly regen + open a `claude/` docs PR if drift); `vercel-prune` (scheduled dry-run
  report → `--apply` gated); `babysit-pr` (watch a PR → retry flaky CI → surface conflicts → advisory).
- **Out of v1:** anything that auto-merges or auto-deploys; email delivery; a second deploy-verify routine
  (D stays OUT). PR-babysit stays **advisory**, never a required check (the standing routines discipline).
- **Risk:** low–medium (prune `--apply` and any PR action touch shared infra → those stories are the ones to
  tier carefully; keep destructive ops dry-run-first + human-confirmed, per blog category 9).
- **Acceptance:** a Telegram standup lands each morning and a recap each week, both delta-accurate against a
  hand-check; `BUILD-ORDER.md` guard stops going red overnight; stale previews trend down.
- **Cap note:** budget the new scheduled routines against Pro = 5/day scheduled (B-smoke already 1/night).
  Prefer **one nightly "ops" routine** that runs several skills over many separate scheduled routines.

### Initiative C — Model split (Opus plans · Sonnet 5 builds, with escalation)

- **In v1:** one edit to `WAYS-OF-WORKING.md` *Model tiers* naming Sonnet 5 + the "escalate, don't guess"
