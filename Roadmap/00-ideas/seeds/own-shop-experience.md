Epic: Absolute White-Label Storefront Experience ("Own Shop")
📋 Epic Overview
As a scaling merchant/tenant,
I want my live custom domain to serve a completely isolated, fully white-labeled storefront experience across every single buyer touchpoint,
So that I can build uncompromised brand equity, establish deep consumer trust, and eliminate any visual trace of the underlying marketplace infrastructure.
🎯 Strategic Orientation & High-Level Heuristics
The moment a custom domain goes live (🟢 active), the platform must become entirely invisible. Platforms like Shopify and Webflow set a high bar, but they still occasionally leak system footprints in checkout subdomains, default asset URLs, or confirmation redirects. To deliver a top-shelf experience, we must protect the merchant’s brand identity ruthlessly.
In e-commerce ecosystems—especially across emerging markets where digital checkout friction is high—any unexpected domain hopping during a transaction triggers security alarms for the buyer. If a customer browses mitienda.mx and suddenly sees bonsai-platform.com/checkout in their address bar, conversion rates collapse due to fraud anxiety.
🧠 Storefront UX Heuristics
• The Illusion of Isolation: Every single link, image path, transaction step, and customer portal page must resolve under the tenant's domain. The platform should act as raw material, not a visible wrapper.
• Aggressive Boundary Enforcement: If a buyer modifies a URL manually or attempts to access a legacy platform path, the system must gracefully capture and translate that request into the merchant's custom domain context.
• Zero-Downtime Migration: Transitioning the public-facing storefront from the platform's shared space to the custom domain must feel instantaneous to the outside world, preserving existing consumer carts and active sessions.
🛠️ Acceptance Criteria
1. Absolute Brand Isolation & White-Label Routing
• AC 1.1: Once the custom domain is active, all core storefront paths (Homepage, Collection pages, Product Detail Pages, Cart, and Static Pages) must render natively and exclusively under the tenant's custom domain.
• AC 1.2: The buyer must never see the underlying platform slug (/s/[slug] or [tenant].platform.com) in the browser address bar, browser status hover-bars, or source code meta tags.
• AC 1.3: All media assets, product images, uploaded custom print files, and document downloads rendered on the storefront must be served from a masked or clean URL structure that aligns with the custom domain identity.
2. Monolithic Checkout & Customer Portal Session
• AC 2.1: The entire checkout funnel—from cart submission through payment processing to the final order success page—must remain strictly on the tenant's custom domain or authenticated custom subdomain (e.g., checkout.mitienda.mx).
• AC 2.2: Customer account creation, login screens, order history tracking, and unified support inbox widgets must match the storefront domain precisely to prevent cross-origin tracking or cookie blocks on modern mobile browsers.
• AC 2.3: Any transactional communication generated by a storefront action (e.g., SMS alerts, WhatsApp updates, or automated order confirmation emails) must reference the custom domain exclusively in its copy and return links.
3. SEO Continuity & Legacy Redirects
• AC 3.1: The system must implement immediate, automated 301 Move Permanently routing rules for any traffic hitting the old marketplace URL structure. Example: A buyer clicking a legacy link to platform.com/s/mitienda/productos/custom-print must be seamlessly routed to mitienda.mx/productos/custom-print without dropping their active shopping cart contents.
• AC 3.2: The storefront engine must dynamically rewrite all robots.txt, sitemap.xml, and open-graph canonical tags (og:url) to point to the new custom domain, instantly signaling search engine crawlers to index the independent brand identity and avoid duplicate content penalties.
4. Tenant Transition & Fail-Safe Mechanics
• AC 4.1: The implementation must provide a clean toggle or automated handler for when a merchant intentionally disconnects or changes their domain.
• AC 4.2: If a domain is removed, the storefront must instantly revert back to the fallback platform route (/s/[slug]) smoothly so the store never displays a raw cloud platform error or breaks down completely.
🔒 Security, Trust & Edge Cases
• Consistent Security Badging: Because modern browsers flag mixed content heavily, the system must guarantee that SSL encryption coverage extends to all customer input areas, including personalization text boxes and dynamic variant selectors, keeping the "Secure Lock" icon unbroken.
• Mobile Layout Fluidity: Brand navigation headers and utility bars must adapt flawlessly on mobile web views when rendering under a custom domain, ensuring that long domain strings do not distort structural header layouts.
• Payload Agnosticism: The routing layers must pass custom configuration inputs (like personalized print text strings) through the rewritten domain layers securely without altering data integrity or drop rates.
