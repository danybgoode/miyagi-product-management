## What went well
- **Reuse over rebuild, epic-wide.** S1's table reused the bulk-import staging pattern's shape; S3's bulk pipeline reused S1's filter query; S4's margin columns reused the profit-analyzer scaffold wholesale; S6's seller shell extraction reused S5's already-shipped nav/badge logic verbatim. Nothing in six sprints built a parallel system where an existing seam could carry the weight.
- **Two-layer review (cross-agent advisory + an independent fresh-reviewer pass) earned its cost repeatedly, not just once.** S3's independent pass caught a real IDOR the cross-agent pass's narrower pattern-match missed. S4's passes caught a deploy-ordering hazard that would have broken an already-live feature. S6's independent pass caught a real scope gap in a guard extension (below) that both a self-review and the cross-agent pass missed. The two layers consistently found *different* classes of bug, not overlapping ones — worth keeping both, not consolidating to one.
- **The live smoke test is itself a test of the code, not just a checkbox.** S3's Daniel-run smoke surfaced a second, independent production incident (the null-slot crash) that no amount of pre-merge review had caught, fixed same-day. Treat the smoke as an adversarial pass, not a rubber stamp.
- **Planning discipline scaled to the risk.** S6.1's shared-`layout.tsx` touch got a dedicated plan-mode session + a cross-agent planning-panel run *before any code existed* — and the panel's one real finding (a duplicate Medusa ownership call) was cheaper to catch in the design than in a PR diff.

## What we learned
- **A grep for the FILE being renamed/deleted doesn't find every reference to what that file represented — you also need to grep the literal VALUE it carried.** S6.2's `Canal.tsx` → `canal-propio` + `apoyo` split correctly swept every code IMPORT of the old file (confirmed via grep, zero missed), but missed two spec files hardcoding the taxonomy SLUG STRING `'canal'` and a flag COUNT that the new flag bumped — both invisible to a filename-scoped grep, both caught only by CI. *Generalizable playbook addition for the next slug rename or registry-size bump: after the file-level sweep, separately grep the literal string/count value across the whole repo, not just the directory being restructured.*
- **A line-cap guard's scope needs to follow the feature, not just the directory it started in.** The anti-monolith guard was built to watch `SETTINGS_DIR`; when S6.2 split `Canal.tsx` and the federation half moved to a new sibling directory (`canal-propio/`, deliberately OUTSIDE `SETTINGS_DIR` since it's no longer a settings section), the guard silently stopped watching it — and the new file immediately re-grew past the old cap (977 lines) with nothing catching it until an independent review pass noticed by hand. *A guard tied to a fixed directory should be re-scoped explicitly whenever a covered feature's files move outside that directory, not left to implicitly "graduate" out of coverage.* Fixed by both decomposing the file (extract, don't just raise the cap — the established precedent held) AND adding the new directory as a second scan root, so the same class of regression can't recur silently.
- **A next-free/auth-free "pure predicate" convention (established by `lib/seller-mode.ts`) needs to be enforced at the FILE level, not the function level.** S6.1's new `isSellShellCandidatePath` was originally a pure function correctly free of side effects, but it lived in the same file as `sellShellEligible()`, which imports `server-only`/`@clerk/nextjs/server` — and importing that file from a Playwright `api` spec failed with an opaque module-resolution error inside Clerk's package, even though the predicate itself touched nothing Clerk-related. The existing `lib/seller-mode.ts` doc comment already states the *reason* for this discipline (so the api spec can load it directly); this epic's gap was applying that reason inconsistently to a sibling file. *When adding a new pure predicate alongside an async/auth-touching sibling, split them into separate files from the start — don't discover the import-poisoning the hard way via a cryptic spec failure.*
- **"Merge on green" from Daniel doesn't mean skip the review layers — it means proceed through them without re-asking at each step.** S6 ran the full two-layer review (cross-agent + independent) even under an explicit merge-on-green authorization, and both layers still found real things worth fixing pre-merge. The authorization changed who decides to stop for a check-in, not whether the checks themselves run.
